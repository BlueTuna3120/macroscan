# Architecture

MacroScan is a **single-file vanilla JS PWA** (`index.html`, ~1750 lines). No build tools, no framework, no bundler. All HTML, CSS, and JavaScript live inline.

---

## File Layout

```
macroscan/
├── index.html              ← entire app (HTML + CSS + JS)
├── sw.js                   ← service worker (cache-first, offline support)
├── site.webmanifest        ← PWA manifest
├── user-data.json          ← auto-generated by sync; all users' data (not committed manually)
├── README.md
├── icons/
│   ├── make-icons.html     ← canvas-based icon generator (run once locally)
│   ├── icon-192.png        ← generated; commit after running make-icons.html
│   ├── icon-512.png
│   └── icon-180.png
└── docs/
    ├── ARCHITECTURE.md     ← this file
    ├── SYNC.md             ← auto-sync design
    ├── BUGS.md             ← audit log
    └── NUTRITION_LOGIC.md  ← BMR/TDEE/macro math
```

## Global State (`S`)

One mutable object holds all runtime state. Mutate fields, then call `render()`.

```js
const S = {
  // Navigation
  screen:      'users' | 'setup' | 'summary' | 'search' | 'history' | 'weight',

  // Multi-user
  currentUser: string,          // active username, '' if none selected
  users:       string[],        // all known usernames

  // Per-user (reloaded on switchUser)
  unit:        'imperial' | 'metric',
  foodUnit:    'g' | 'oz' | 'ml' | 'floz' | 'srv',
  profile:     { weightLbs, heightFt, heightIn, weightKg, heightCm, age, gender, activity, goal },
  targets:     { calories, protein, carbs, fat } | null,
  entries:     logEntry[],      // today's entries for currentUser only

  // Food lookup / logging transient state
  scannedFood:      foodObject | null,
  foodReady:        boolean,
  foodWeightDisplay: string,    // raw input value
  searchResults:    foodObject[] | null,
  searchLoading:    boolean,
  searchError:      string,
  customFood:       { name, calories, protein, carbs, fat },

  // Scanner
  scannerOpen:      boolean,
  scannerStatus:    string,
  scannerError:     string,
  scannerManualCode: string,
  scannerActive:    boolean,
  zxingLoaded:      boolean,

  // Weight screen
  weightInput:      string,

  // Sync state (drives header indicator)
  _syncState:  'idle' | 'pending' | 'syncing' | 'ok' | 'error',

  // Stale-response guards
  _lastLookupId: number,
  _searchId:     number,
};
```

## localStorage Schema

All keys are namespaced by username. Global keys (not per-user) use no suffix.

### Global keys

| Key | Value |
|---|---|
| `ms:users` | JSON string array — all profile names |
| `ms:current_user` | String — last active username |
| `ms:gh_token` | String — GitHub PAT for push (never synced to repo) |

### Per-user keys (`{u}` = username)

| Key | Value |
|---|---|
| `ms:profile:{u}` | JSON — profile object |
| `ms:targets:{u}` | JSON — `{ calories, protein, carbs, fat }` |
| `ms:unit:{u}` | `'imperial'` or `'metric'` |
| `ms:foodUnit:{u}` | `'g'`, `'oz'`, `'ml'`, `'floz'`, `'srv'` |
| `ms:entries:{u}:YYYY-MM-DD` | JSON array of log entries for that date |
| `ms:dates:{u}` | JSON array of date strings that have entries (newest first) |
| `ms:weightlog:{u}` | JSON array of weight entries |
| `ms:modified:{u}` | Epoch ms string — timestamp of last local save (used by sync merge) |

## Render Cycle

```
mutate S  →  call render()  →  innerHTML rebuild of #content
```

`render()` rebuilds the entire `#content` div on every state change. No virtual DOM or diffing.

**Exception — food result screen:** When `amountInput` is focused, `renderFoodResult()` is skipped to prevent iOS from collapsing the keyboard. `updateAmountPreview()` is called directly from the input's `oninput` handler instead, updating only the preview row nodes in-place.

The nav bar and scanner modal are managed outside `#content` and updated separately.

### Screen routing

| `S.screen` | Renderer | Notes |
|---|---|---|
| `'users'` | `renderUsers()` | Shown on first load or when switching profiles. Nav bar hidden. |
| `'setup'` | `renderSetup()` | Profile form + settings. Reached after creating a user or from Settings tab. |
| `'summary'` | `renderSummary()` | Today's dashboard. Default screen when targets are set. |
| `'search'` | `renderSearch()` | Food lookup. Nav tab "Today" stays active. |
| `'search'` + `S.foodReady` | `renderFoodResult()` | Amount entry + add-to-log. Same screen value as search. |
| `'history'` | `renderHistory()` | Expandable per-day log. |
| `'weight'` | `renderWeight()` | Weight tracker + SVG chart. |

## Multi-User System

### User lifecycle

```
renderUsers()
  → createUser(name)       new user → setup screen
  → switchUser(username)   existing user → summary or setup

renderSetup()
  → "Switch Profile" link  → renderUsers()
  → "Delete profile" link  → deleteCurrentUser() → renderUsers()
```

`switchUser()` calls `_loadUserData(username)` which reads all per-user localStorage keys into `S` and sets `S.screen` to `'summary'` if targets exist or `'setup'` if not.

### Legacy migration

On first load after the multi-user update, if `ms:profile` exists (old single-user key) but `ms:users` does not, `_migrateLegacy()` runs automatically. It creates a profile named `"Me"`, copies all old keys to the namespaced equivalents, and deletes nothing (old keys remain as dead entries).

## Data Flow

### Profile → Targets

```
User inputs (weight, height, age, sex, activity, goal)
  → BMR via Mifflin-St Jeor
  → × activity multiplier = TDEE
  → + goal calorie offset
  → computeTargets(TDEE, goal, weightLbs) → { calories, protein, carbs, fat }
  → S.targets = result
  → save() → S.screen = 'summary'
```

See `docs/NUTRITION_LOGIC.md` for the full math.

### Food Logging

```
Search / barcode scan / custom entry
  → select food  (all nutriment values stored per 100g)
  → user picks amount + unit
  → unitToGrams(amount, unit) → grams
  → ratio = grams / 100
  → scale all macros by ratio
  → push to S.entries
  → save()  →  schedulePush()  →  4 s debounce  →  _silentPush()
```

### save() → Auto Sync

Every `save()` call:
1. Writes all per-user localStorage keys
2. Stamps `ms:modified:{u}` with `Date.now()`
3. Calls `schedulePush()` — resets a 4-second debounce timer
4. After 4 s of no further saves: `_silentPush()` writes `user-data.json` to the repo

See `docs/SYNC.md` for full sync design.

## Network Calls

| Purpose | Endpoint | Auth | Guard |
|---|---|---|---|
| Food text search | `world.openfoodfacts.org/cgi/search.pl` | None | `_searchId` stale-response guard |
| Barcode lookup | `world.openfoodfacts.org/api/v2/product/:code` | None | `_lastLookupId` stale-response guard |
| Sync push (GET SHA) | `api.github.com/repos/{owner}/{repo}/contents/user-data.json` | PAT | Runs before PUT |
| Sync push (PUT) | same | PAT | Only if token present |
| Sync pull (load) | `./user-data.json?t={timestamp}` | None | Cache-busted; SW excluded |

Food search uses only `_100g` nutriment fields. Per-serving API data is not used (too inconsistent across products).

---

## Service Worker (`sw.js`)

Strategy: **cache-first** for same-origin GET requests, **network pass-through** for everything else.

- External requests (Open Food Facts, ZXing CDN, Google Fonts) are never intercepted
- `user-data.json` is explicitly excluded — always fetched fresh from network
- Static app shell (`index.html`, `site.webmanifest`, icons) is pre-cached on install
- Cache name is versioned (`macroscan-v1.1`) — bumping it on deploy purges the old cache

## Security Notes

- `escHtml()` sanitizes all user-controlled strings before `innerHTML` insertion. Any new template literals that include user data must use it.
- The GitHub PAT (`ms:gh_token`) is stored in `localStorage` only — it is never included in `buildPayload()` and never written to `user-data.json` or any synced file.
- `user-data.json` in the repo is publicly readable (public repo + GitHub Pages). Do not store anything sensitive. This is by design for this use case.
- No server, no backend, no auth layer beyond the GitHub token for writes.

## Known Architectural Limits

- **Full innerHTML rebuild on every state change.** Works at this scale; would need partial update strategy if the app grows significantly. `updateAmountPreview()` is the one targeted exception.
- **Global scope.** All functions and `S` are global. Fine for a single-file app.
- **Single `TODAY` constant.** Computed once at boot. If the app stays open across midnight without a reload, `TODAY` will be stale until the page is refreshed. Entries logged after midnight will go to the previous day's key until reload.
- **`user-data.json` is one file for all users.** Concurrent pushes from two devices within the same debounce window could cause a lost update (last write wins for the whole file). Acceptable for a small personal group; would need conflict resolution for broader use.
